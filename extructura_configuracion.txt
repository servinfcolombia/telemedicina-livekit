## 1️⃣ Prompts Personalizados (`.opencode/prompts/`)

### 📄 `.opencode/prompts/fhir-extraction.txt`
```txt
# Prompt para Extracción de Entidades FHIR con LLM
# Versión: 1.0 | Idioma: Español médico | Output: JSON FHIR R4 válido

## ROL DEL SISTEMA
Eres un asistente clínico especializado en codificación médica y estándares HL7 FHIR R4.
Tu única función es extraer entidades clínicas de transcripciones de consultas médicas
y estructurarlas como recursos FHIR válidos.

## REGLAS CRÍTICAS (NO NEGOCIABLES)

### 1. Principio de No Invención
- NUNCA inventes diagnósticos, medicamentos o datos que no estén explícitamente mencionados
- Si la información es ambigua, usa "unknown" o omite el campo, nunca asumas
- Para síntomas: solo codifica si hay descripción clara de intensidad, ubicación o duración

### 2. Formato de Salida Obligatorio
- Output SIEMPRE en formato JSON válido (no Markdown, no texto explicativo)
- Estructura: Bundle FHIR R4 de tipo "collection"
- Cada recurso debe incluir: resourceType, id (generado), meta.profile (URL del perfil FHIR)

### 3. Codificación Estándar
Usa exclusivamente estos sistemas de codificación:
- Diagnósticos/Condiciones: SNOMED-CT (http://snomed.info/sct)
- Medicamentos: RxNorm (http://www.nlm.nih.gov/research/umls/rxnorm)
- Observaciones/Síntomas: LOINC (http://loinc.org)
- Procedimientos: SNOMED-CT o CPT según contexto

### 4. Recursos FHIR Permitidos
Solo genera estos tipos de recursos:
- Patient (solo si hay datos demográficos nuevos)
- Encounter (la consulta actual)
- Condition (diagnósticos, problemas activos)
- Observation (signos vitales, síntomas reportados)
- MedicationRequest (prescripciones)
- Procedure (procedimientos realizados)
- DocumentReference (referencia a grabación de audio)

## 3️⃣ Scripts de Automatización

### 📄 `.opencode/hooks/on-phase-complete.sh`
```bash
#!/bin/bash
#==============================================================================
# Hook: on-phase-complete.sh
# Propósito: Actualizar progreso del proyecto y notificar al equipo
# Uso: Ejecutado automáticamente por OpenCode al completar una fase
#==============================================================================

set -euo pipefail

# Parámetros recibidos de OpenCode
PHASE_ID="${1:-unknown}"
PHASE_NAME="${2:-Unknown Phase}"
STATUS="${3:-unknown}"  # success, failed, partial
TIMESTAMP=$(date -Iseconds)
USER="${OPENCODE_USER:-$(whoami)}"

# Configuración
PROGRESS_FILE=".opencode/PROGRESS.md"
NOTIFICATION_CHANNEL="${NOTIFICATION_CHANNEL:-slack}"  # slack, email, webhook
LOG_FILE="logs/hooks/phase-complete-$(date +%Y%m%d).log"

# Crear directorio de logs si no existe
mkdir -p "$(dirname "$LOG_FILE")"

# Función de logging
log() {
  echo "[$(date -Iseconds)] $*" | tee -a "$LOG_FILE"
}

# Función para actualizar PROGRESS.md
update_progress() {
  local phase_id="$1"
  local status="$2"
  local timestamp="$3"
  
  if [[ ! -f "$PROGRESS_FILE" ]]; then
    log "⚠️  $PROGRESS_FILE no encontrado, creando plantilla..."
    cat > "$PROGRESS_FILE" << 'EOF'
# 🏥 Telemedicina LiveKit - Progreso del Proyecto

| Fase | ID | Estado | Completado | Validado | Notas |
|------|-----|--------|------------|----------|-------|
EOF
  fi
  
  # Determinar emoji y texto de estado
  local emoji=""
  local status_text=""
  case $status in
    success) emoji="✅"; status_text="Completada" ;;
    failed) emoji="❌"; status_text="Fallida" ;;
    partial) emoji="⚠️"; status_text="Parcial" ;;
    *) emoji="⏳"; status_text="En progreso" ;;
  esac
  
  # Actualizar o agregar fila en la tabla
  if grep -q "| $phase_id |" "$PROGRESS_FILE" 2>/dev/null; then
    # Actualizar fila existente
    sed -i "s|.*| $phase_id |.*|✅ $phase_id | $status_text | $timestamp | $([ "$status" = "success" ] && echo "✔️" || echo "❌") | Fase $status_text: $PHASE_NAME |" "$PROGRESS_FILE"
  else
    # Agregar nueva fila
    echo "| $emoji $PHASE_NAME | $phase_id | $status_text | $timestamp | $([ "$status" = "success" ] && echo "✔️" || echo "❌") | $([ "$status" = "failed" ] && echo "Revisar logs/" || echo "") |" >> "$PROGRESS_FILE"
  fi
  
  log "📝 Progreso actualizado en $PROGRESS_FILE"
}

# Función para notificar al equipo
notify_team() {
  local phase_name="$1"
  local status="$2"
  local user="$3"
  
  local message=""
  local color=""
  
  case $status in
    success)
      message="✅ Fase completada: *$phase_name* por @$user"
      color="good"
      ;;
    failed)
      message="❌ Fase fallida: *$phase_name* por @$user - Revisar logs/"
      color="danger"
      ;;
    partial)
      message="⚠️ Fase parcial: *$phase_name* por @$user - Algunas tareas pendientes"
      color="warning"
      ;;
  esac
  
  case $NOTIFICATION_CHANNEL in
    slack)
      if [[ -n "${SLACK_WEBHOOK_URL:-}" ]]; then
        curl -s -X POST -H 'Content-type: application/json' \
          --data "{\"text\":\"$message\",\"attachments\":[{\"color\":\"$color\"}]}" \
          "$SLACK_WEBHOOK_URL" > /dev/null 2>&1 && \
          log "🔔 Notificación enviada a Slack" || \
          log "⚠️  Falló envío a Slack (verificar SLACK_WEBHOOK_URL)"
      else
        log "⚠️  SLACK_WEBHOOK_URL no configurado, omitiendo notificación Slack"
      fi
      ;;
    email)
      if command -v mail &> /dev/null && [[ -n "${TEAM_EMAIL:-}" ]]; then
        echo "$message" | mail -s "Telemedicina LiveKit: $phase_name - $status" "$TEAM_EMAIL" && \
        log "🔔 Notificación enviada por email a $TEAM_EMAIL" || \
        log "⚠️  Falló envío de email"
      else
        log "⚠️  TEAM_EMAIL no configurado o mail no disponible, omitiendo notificación email"
      fi
      ;;
    webhook)
      if [[ -n "${WEBHOOK_URL:-}" ]]; then
        curl -s -X POST -H 'Content-type: application/json' \
          --data "{\"phase\":\"$phase_name\",\"status\":\"$status\",\"user\":\"$user\",\"timestamp\":\"$timestamp\"}" \
          "$WEBHOOK_URL" > /dev/null 2>&1 && \
        log "🔔 Notificación enviada a webhook" || \
        log "⚠️  Falló envío a webhook"
      else
        log "⚠️  WEBHOOK_URL no configurado, omitiendo notificación webhook"
      fi
      ;;
    *)
      log "ℹ️  Canal de notificación '$NOTIFICATION_CHANNEL' no reconocido"
      ;;
  esac
}

# Función para generar resumen de la fase
generate_phase_summary() {
  local phase_id="$1"
  local output_file="logs/phases/${phase_id}-summary-$(date +%Y%m%d-%H%M).md"
  
  mkdir -p "$(dirname "$output_file")"
  
  cat > "$output_file" << EOF
# 📋 Resumen de Fase: $PHASE_NAME

**ID:** $phase_id  
**Estado:** $STATUS  
**Completado por:** $USER  
**Timestamp:** $TIMESTAMP  

## Archivos Generados
\`\`\`bash
$(git diff --name-only --diff-filter=A HEAD~1 HEAD 2>/dev/null | grep -E "^(${phase_id//-/_}|agents|frontend|backend)/ | head -20 || echo "No se detectaron cambios vía git")
\`\`\`

## Validaciones Ejecutadas
\`\`\`bash
# Comandos de validación ejecutados (revisar logs de OpenCode)
# Para detalles completos: ver sesión de OpenCode o logs/
\`\`\`

## Próximos Pasos
$(case $STATUS in
  success) echo "- ✅ Fase lista para revisión manual" ;;
  failed) echo "- ❌ Revisar errores en logs/ y reintentar" ;;
  partial) echo "- ⚠️ Completar tareas pendientes antes de avanzar" ;;
esac)

---
*Generado automáticamente por on-phase-complete.sh*
EOF
  
  log "📄 Resumen generado: $output_file"
}

# Función para verificar dependencias de la siguiente fase
check_next_phase_deps() {
  local current_phase="$1"
  
  # Mapeo de fases y sus dependencias
  declare -A NEXT_PHASE_DEPS=(
    ["phase-1-infra"]="docker,kubectl,helm"
    ["phase-2-backend"]="python3.10+,pip,pytest"
    ["phase-3-frontend"]="node18+,npm,playwright"
    ["phase-4-ia"]="python3.10+,torch,whisper.cpp"
    ["phase-5-security"]="kubectl,trivy,axe-core"
    ["phase-6-deploy"]="helm,k6,aws-cli"
  )
  
  local next_phase=""
  case $current_phase in
    phase-1-infra) next_phase="phase-2-backend" ;;
    phase-2-backend) next_phase="phase-3-frontend" ;;
    phase-3-frontend) next_phase="phase-4-ia" ;;
    phase-4-ia) next_phase="phase-5-security" ;;
    phase-5-security) next_phase="phase-6-deploy" ;;
    *) return 0 ;;  # Última fase o desconocida
  esac
  
  if [[ -n "${NEXT_PHASE_DEPS[$next_phase]:-}" ]]; then
    log "🔍 Verificando dependencias para $next_phase..."
    local missing_deps=()
    
    for dep in ${NEXT_PHASE_DEPS[$next_phase]//,/ }; do
      if ! command -v "${dep%%+*}" &> /dev/null; then
        missing_deps+=("$dep")
      fi
    done
    
    if [[ ${#missing_deps[@]} -gt 0 ]]; then
      log "⚠️  Dependencias faltantes para $next_phase: ${missing_deps[*]}"
      echo "⚠️  Dependencias faltantes para $next_phase: ${missing_deps[*]}" >> "$PROGRESS_FILE"
    else
      log "✅ Todas las dependencias para $next_phase están instaladas"
    fi
  fi
}

#==============================================================================
# EJECUCIÓN PRINCIPAL
#==============================================================================
main() {
  log "🔄 Hook on-phase-complete ejecutado"
  log "   Fase: $PHASE_NAME ($PHASE_ID)"
  log "   Estado: $STATUS"
  log "   Usuario: $USER"
  
  # 1. Actualizar archivo de progreso
  update_progress "$PHASE_ID" "$STATUS" "$TIMESTAMP"
  
  # 2. Generar resumen de la fase
  if [[ "$STATUS" != "unknown" ]]; then
    generate_phase_summary "$PHASE_ID"
  fi
  
  # 3. Notificar al equipo si la fase es crítica
  if [[ "$STATUS" != "unknown" ]] && [[ "$PHASE_ID" =~ ^(phase-1-infra|phase-5-security|phase-6-deploy)$ ]]; then
    notify_team "$PHASE_NAME" "$STATUS" "$USER"
  fi
  
  # 4. Verificar dependencias para la siguiente fase (solo si éxito)
  if [[ "$STATUS" == "success" ]]; then
    check_next_phase_deps "$PHASE_ID"
  fi
  
  # 5. Si falló, sugerir rollback
  if [[ "$STATUS" == "failed" ]]; then
    log "❌ Fase fallida. Para rollback:"
    log "   git revert --no-commit \$(git log -1 --format=%H -- ${PHASE_ID//-/_}/) && git commit -m 'Rollback $PHASE_ID'"
  fi
  
  log "✅ Hook completado"
}

# Ejecutar
main "$@"
```

## 🔄 Rotación de Credenciales

### Frecuencia Recomendada
| Tipo de Credencial | Frecuencia de Rotación | Método |
|-------------------|----------------------|--------|
| JWT Signing Key | Cada 90 días | Generar nuevo, mantener anterior 24h para transición |
| API Keys (LiveKit, OpenEMR) | Cada 180 días | Crear nueva en proveedor, actualizar config, eliminar anterior |
| Database Passwords | Cada 90 días | Usar script de rotación con downtime mínimo |
| SSL Certificates | Cada 90 días (Let's Encrypt) o 1 año (comercial) | Renovación automática con cert-manager |

### Script de Rotación de JWT Secret
```bash
#!/bin/bash
# scripts/rotate-jwt-secret.sh

set -e

echo "🔄 Rotando JWT secret..."

# 1. Generar nuevo secret
NEW_SECRET=$(openssl rand -base64 64)

# 2. Actualizar en Vault/Kubernetes
kubectl create secret generic jwt-secrets \
  --from-literal=current=$NEW_SECRET \
  --from-literal=previous=$(kubectl get secret jwt-secrets -o jsonpath='{.data.current}' | base64 -d) \
  --dry-run=client -o yaml | kubectl apply -f -

# 3. Reiniciar servicios que usan JWT (rolling update)
kubectl rollout restart deployment -n telemedicina -l app=backend

# 4. Esperar propagación (JWT anterior válido por 24h)
echo "✅ Nuevo JWT secret aplicado. Secret anterior válido hasta $(date -d '+24 hours')"

# 5. Programar eliminación del secret anterior (opcional)
# cron: eliminar previous después de 25 horas
```

### 📄 `tests/data/medical-stt-dataset/README.md`
```markdown
# 🗣️ Dataset Médico para Validación de STT

## Propósito
Validar que el modelo Whisper.cpp transcribe con precisión ≥95% (WER ≤0.05) 
audio de consultas médicas en español, incluyendo terminología clínica.

## Estructura del Dataset
```
tests/data/medical-stt-dataset/
├── audio/
│   ├── consult-001.wav  # 2:15 min, dolor abdominal, 48kHz mono
│   ├── consult-002.wav  # 3:42 min, seguimiento hipertensión
│   ├── consult-003.wav  # 1:58 min, prescripción antibiótico
│   └── ... (20 archivos totales)
├── transcripts/
│   ├── consult-001.txt  # Transcripción de referencia (gold standard)
│   ├── consult-002.txt
│   └── ...
├── metadata.json        # Información de cada archivo
└── evaluate.py          # Script para calcular WER
```

## Características del Audio
- **Formato**: WAV, 48kHz, 16-bit, mono (optimizado para STT)
- **Duración**: 1-5 minutos por archivo (representativo de consultas reales)
- **Contenido**: 
  - Voz de médico y paciente (turn-taking)
  - Terminología médica en español: síntomas, diagnósticos, medicamentos
  - Variación de acentos: México, Colombia, Argentina, España
  - Condiciones de grabación: oficina clínica (ruido de fondo bajo)

## Métricas de Evaluación
```python
# Calcular Word Error Rate (WER)
WER = (S + D + I) / N
# Donde:
# S = Sustituciones (palabras incorrectas)
# D = Deleciones (palabras omitidas)
# I = Inserciones (palabras añadidas)
# N = Total de palabras en referencia

# Umbral de aceptación: WER ≤ 0.05 (95% precisión)
```

## Ejecutar Evaluación
```bash
# 1. Transcribir con Whisper.cpp
cd agents
python -m whisper transcribe \
  --model medium \
  --language es \
  --audio ../tests/data/medical-stt-dataset/audio/*.wav \
  --output-dir /tmp/stt-output

# 2. Calcular WER
python ../tests/data/medical-stt-dataset/evaluate.py \
  --hypothesis /tmp/stt-output \
  --reference ../tests/data/medical-stt-dataset/transcripts

# 3. Ver resultado
# Expected: WER: 0.042 (pasa) o WER: 0.067 (falla)
```

## Contribuir al Dataset
Para agregar nuevos casos de prueba:
1. Grabar consulta real (con consentimiento) o sintetizar con TTS médico
2. Transcribir manualmente con doble verificación por médico
3. Anonimizar PHI: reemplazar nombres, IDs, fechas específicas
4. Agregar a `metadata.json` con etiquetas: especialidad, complejidad, acento
5. Ejecutar `evaluate.py` para asegurar que no degrada WER promedio

## Consideraciones Éticas
- Todos los audios deben tener consentimiento informado para uso en desarrollo
- Anonimización estricta: eliminar cualquier dato que pueda identificar pacientes
- Almacenamiento seguro: cifrado en reposo, acceso restringido al equipo de IA
- Retención: eliminar audios originales después de 2 años, conservar solo transcripts anonimizados
```
